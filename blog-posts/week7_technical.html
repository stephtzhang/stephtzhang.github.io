Optimizing SQL boils down to a straight-forward concept: reducing the amount of data that a query accesses. Following simple best practices can help achieve that.

Index

A table without indexing is like library that doesn't follow the Dewey Decimal system. Using a PRIMARY KEY with AUTO_INCREMENT provides a path for data access. Instead of checking every row until a piece of data is found, a query can hone in on the area of the index tree where a PRIMARY KEY lives.

Avoid SELECT *

SELECT * requires a query to iterate through every column in a row. Specifying the columns that you'd like to access siginificantly reduces the amount of data touched.

LIMIT 1 for unique rows

If you are searching for a single row, appending LIMIT 1 to the end of a query will tell your database to stop scanning after it has located the row.

Use NOT NULL

Setting columns to NOT NULL avoids the extra space and complexity required for NULL attributes. In cases where you need NULL attributes, query calls should specify whether NULLs should be included in the result.

EXPLAIN queries

Adding EXPLAIN before a SELECT query will display a table showing the query's stats (the number of rows that were accessed, the keys in use, etc). This is not a best practice per se, but it can help locate the source of slow queries, so that best practices can then be implemented.
